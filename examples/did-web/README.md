# Example did:web Documents for SimpulseID

This folder contains example **did:web DID documents** for entities used in the
SimpulseID / ENVITED identity ecosystem. These examples demonstrate how program,
participant, administrator, and user identifiers can be published under the
`did.identity.ascs.digital` domain.

All documents in this folder are **examples only** and contain placeholder JWS
signatures (`EXAMPLE_SIGNATURE_PAYLOAD_*`). When deploying these DIDs, replace
the placeholder signatures with real JWS signatures generated by the controller.

---

## Program DID Documents

The ASCS membership programs have stable did:web identifiers:

did:web:did.identity.ascs.digital:programs:ascs-base-membership
did:web:did.identity.ascs.digital:programs:ascs-envited-membership

They must be hosted at:

<https://did.identity.ascs.digital/programs/ascs-base-membership/did.json>
<https://did.identity.ascs.digital/programs/ascs-envited-membership/did.json>

The example files in this folder show:

- the expected JSON-LD structure for program DIDs,
- the required verification methods (Tezos and Etherlink/EVM),
- and the placeholder JWS values that must be replaced when signing.

Only update:

- the JWS value in `proof.jws`
- any `serviceEndpoint` or `alsoKnownAs` URLs if your hosting structure changes.

---

## Participant, Administrator, and User DIDs

Additional did:web examples in this folder illustrate:

- Participant DIDs  
  (`did:web:did.identity.ascs.digital:participants/...`)

- Administrator DIDs  
  (`did:web:did.identity.ascs.digital:users/ascs/admin-<opaque-id>`)

- User DIDs  
  (`did:web:did.identity.ascs.digital:users/<participant>/u-<opaque-id>`)

These documents contain:

- multiple verification keys (Tezos + Etherlink)
- no personal data
- placeholder JWS signatures to be replaced before deployment

They demonstrate how credential subjects in the example credentials can be linked
to did:web identifiers without exposing private information.

---

## Deployment Notes

To deploy these DIDs in production:

1. Place each file at its required `https://did.identity.ascs.digital/.../did.json` path.
2. Replace `EXAMPLE_SIGNATURE_PAYLOAD_*` with real signatures.
3. Ensure that hosting preserves:
   - correct MIME type (`application/did+json`)
   - correct HTTPS URL structure
4. Ensure that controllers retain all private keys used to generate signatures.
